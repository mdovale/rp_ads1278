## Handoff: FPGA status and remaining bring-up work

**Type:** Status + scope
**Priority:** High

### Summary

The `fpga/` tree is no longer empty. Core RTL, top-level integration, block-design TCL, constraints, and build/deploy scripts are in place. Remaining FPGA work is now about validation, cleanup, and software-facing documentation rather than first-pass implementation.

### Current Implemented State

- `fpga/rtl/red_pitaya_top.sv` instantiates the PS wrapper and `ads1278_axi_slave`, and maps the ADS1278 signals onto E1 P-side expansion pins.
- `fpga/rtl/ads1278_axi_slave.sv` exposes the current register map:
  - `0x00` to `0x1C`: CH1 to CH8
  - `0x20`: `STATUS`
  - `0x24`: `CTRL`
  - `0x28`: `EXTCLK_DIV`
- `fpga/rtl/ads1278_acq_top.v` wraps:
  - `ads1278_spi_tdm.v`
  - `ads1278_extclk_gen.v`
  - `ads1278_sync_pulse.v`
- `fpga/source/system_design_bd_rp125_14/system.tcl` assigns the AXI GP0 register aperture at `0x40000000`, size `0x1000`.
- `fpga/source/cons_rp125_14/ports.xdc` constrains the README signal mapping on E1:
  - `exp_p_io[0]` G17 SCLK
  - `exp_p_io[1]` H16 MISO
  - `exp_p_io[2]` J18 DRDY
  - `exp_p_io[3]` K17 SYNC
  - `exp_p_io[4]` L14 EXTCLK
- `fpga/source/cfg_rp125_14/ads1278.tcl`, `fpga/regenerate_project_and_bd.tcl`, `fpga-build.sh`, and `fpga-deploy.sh` are present and wired for the current project.

### What Is Still Not Proven

- No successful FPGA build artifact is recorded in the repo.
- No hardware validation is recorded for:
  - DRDY-triggered frame capture
  - channel ordering
  - `SYNC` pulse behavior
  - `EXTCLK_DIV` behavior on real hardware
  - overflow behavior
- No software layer exists yet to exercise the MMIO register block from Linux on the board.

### Known Gaps And Risks

- `fpga-build.sh` still expects `fpga/library/lib_src/my_cores_build_src` when core generation is enabled, but only `make_cores.tcl` exists today and it is a no-op placeholder.
- The repository contains a real FPGA implementation but not the server/client layers needed for practical integration testing.
- The checked-in build flow should be verified against the intended Vivado version before treating it as stable.
- `EXTCLK_DIV` is currently reused by the acquisition logic and the EXTCLK generator, so software documentation needs to be explicit about what the divider controls.

### Suggested Next Steps

1. Run a clean FPGA build for `rp125_14` and record the exact command path that works.
2. Decide whether `fpga-build.sh` should default to `--skip-cores` or whether the expected `my_cores_build_src` directory should be restored.
3. Write down the software-facing contract for the register block:
   - register offsets
   - access semantics
   - reset values
   - enable and SYNC behavior
4. Implement the minimal `server/` layer so the register block can be exercised from Linux over `/dev/mem`.
5. After the server exists, verify real hardware behavior and update this handoff or replace it with a bring-up log.

### Success Criteria

- FPGA build works reproducibly from the checked-in scripts.
- Register map and signal ownership are documented clearly enough for the server implementation.
- At least one board-level smoke test confirms that the MMIO registers change as expected during acquisition.
