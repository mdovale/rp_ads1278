# Board IO Wiring

This doc describes the current board-level signal contract between the Red Pitaya FPGA design and the ADS1278EVM-facing connector wiring. It covers the implemented E1 expansion mapping in the RTL and constraints, plus the intentionally unused signals around it.

## Goal

Define which Red Pitaya FPGA pins currently carry the ADS1278 interface, what direction each signal has, and which board-level assumptions the FPGA design makes.

## Scope

- In scope: the E1 expansion mapping, signal directions, package pins, intended ADS1278EVM connections, and current unused connector lines.
- Out of scope: server/client software behavior, MMIO register details, and analog front-end considerations on the ADS1278EVM.

## User-facing behavior

The current FPGA design uses the Red Pitaya E1 expansion connector for all ADS1278-facing digital signals. High-speed acquisition does not use Linux GPIO control paths.

Current implemented mapping:

| Function | Red Pitaya signal | Package pin | Direction | ADS1278EVM side | Notes |
|------|------|------|------|------|------|
| SPI clock | `exp_p_io[0]` | G17 | Output | `SCLK` | Driven by FPGA acquisition logic |
| SPI data in | `exp_p_io[1]` | H16 | Input | `DOUT1` | TDM stream carrying CH1..CH8 |
| DRDY event | `exp_p_io[2]` | J18 | Input | `/DRDY_FSYNC` | Falling edge starts the capture sequence |
| SYNC control | `exp_p_io[3]` | K17 | Output | `/SYNC` | Active-low pulse generated by FPGA |
| ADC clock | `exp_p_io[4]` | L14 | Output | `EXTCLK` | Driven by FPGA clock divider |
| Ground | RP `GND` | N/A | N/A | EVM `GND` | Required common reference |

Current wiring expectations:

- `DIN` is not used and is expected to be strapped low on the EVM.
- `CS` is not used for ADS1278 TDM readout and can remain unconnected.
- Only the E1 P-side pins listed above are used for the ADS1278 path.
- `exp_p_io[5]` through `exp_p_io[7]` and all `exp_n_io[*]` lines are currently left unused by this feature path.

The top-level RTL reflects that contract directly:

- output pins are driven onto `exp_p_io[0]`, `exp_p_io[3]`, and `exp_p_io[4]`
- input pins are left high-Z at the top level and sampled back from `exp_p_io[1]` and `exp_p_io[2]`
- the entire `exp_n_io` bus is driven high-Z

## Architecture

The board IO contract is defined in three places that should stay aligned:

1. `README.md` declares the intended Red Pitaya to ADS1278EVM mapping.
2. `fpga/rtl/red_pitaya_top.sv` applies the logical signal assignment at the top-level RTL boundary.
3. `fpga/source/cons_rp125_14/ports.xdc` binds those logical ports to Red Pitaya package pins and IO standards.

Signal ownership in the current design:

- `ads1278_axi_slave` owns the software-controlled acquisition block and exposes:
  - `sclk_o`
  - `sync_n_o`
  - `extclk_o`
  - `miso_i`
  - `drdy_n_i`
- `red_pitaya_top.sv` maps those signals onto the expansion connector:
  - `ads_sclk` -> `exp_p_io[0]`
  - `ads_sync_n` -> `exp_p_io[3]`
  - `ads_extclk` -> `exp_p_io[4]`
  - `exp_p_io[1]` -> `ads_miso`
  - `exp_p_io[2]` -> `ads_drdy_n`

Constraint behavior in `ports.xdc`:

- all ADS1278-facing E1 pins are constrained as `LVCMOS33`
- output-specific `SLEW FAST` and `DRIVE 8` are applied only to the three output pins
- unused neighboring E1 pins still receive package/IOSTANDARD constraints, but they are not functionally driven by this feature path

## Known risk areas

- The design assumes the ADS1278 path uses E1 PL-connected pins, not PS-side MIO or Linux-controlled GPIOs.
- The `EXTCLK` output is expected to reach an ADS1278EVM SMA clock input through external wiring, which is mechanically awkward and should be treated as a bring-up risk.
- `exp_n_io[*]` pins are constrained but currently unused, so future board-level changes should avoid assuming they carry ADS1278 traffic.
- Any future remap of the E1 connector must update `README.md`, `red_pitaya_top.sv`, and `ports.xdc` together.
